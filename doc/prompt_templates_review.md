# 提示词模板 (Prompt Templates) 审查与翻译报告

通过对 `agent/prompt_templates.py` 的完整代码审查，我为您整理了当前系统中所有核心 Agent 模块的系统提示词（System Prompts）的中文翻译与核心逻辑总结。同时，我也专门针对“是否存在作弊行为”进行了分析。

## 🎯 结论：是否存在“作弊”？

**结论：没有作弊（No Hardcoded Cheating）。**

在所有的提示词中，**没有**发现任何直接针对特定测试用例（如 G1_01、G2_05、G3_12 等）的硬编码答案。系统中所有的示例都是为了向大模型解释**泛化规律**和**物理引擎机制**，例如：

1. **原理解释而非直接给答案**：提示词中解释了“如果目标要求把物品放进容器，必须先判断容器是否 OPEN，如果是 CLOSED 必须先执行 `[open]`”。这是教大模型理解物理引擎规则，而不是告诉它“G2_06必须开柜子”。
2. **通用场景举例**：在 `EXECUTOR_SYSTEM_PROMPT` 的规则 8 中，提到了“例如：指令说‘如果电视开着’，但电视明确标记为‘BROKEN’，这种情况说明前置条件恒假，必须用 `[ask]` 报告”。这里虽然用了“电视”作为例子（这确实与 G3 测试很相关），但它的目的是为了让大模型理解“什么是固有恒假的条件（Inherently False Preconditions）”这一抽象概念，使其能在遇到“微波炉坏了”、“电脑坏了”等不同场景时都能泛化处理，属于合理的 Few-shot Prompting（少样本提示），而非作弊。
3. **严格的物理限制约束**：所有提示词都在极力约束 LLM **不要跳步**（No Magic Macros）。明确禁止了大模型在没有 `[walk]` 的情况下直接 `[grab]`，或者在没有 `[grab]` 的情况下直接 `[putin]`。这恰恰是反作弊的体现。

## 📄 核心提示词模块中文翻译与解析

以下是系统中各核心 Agent 提示词的翻译和功能解析：

### 1. 意图推理引擎 (GOAL_REASONER_SYSTEM_PROMPT)
**职责**：分析用户的原始自然语言指令，提取出深层意图（Deep Intent），并找出可接受的物理替代品。
**核心规则翻译**：
- 必须使用“为什么（Why）”提问法挖掘意图。直到核心意图（如“用户想要解饿”、“用户想要照亮房间”）清晰为止。
- 深层意图**绝对不能**包含具体的物理物品名称（例如：不要说“给我拿个汉堡”，而是说“解饿”）。
- **寻找替代品**：如果确切物品不可用，必须推荐相同物理类别和功能的物品（例如要汉堡，可以推荐三明治）。
- **模糊指令检测**：机器人生活在确定性的物理世界中。如果指令有语义歧义（比如指代不明的“那个东西”，或者“调到合适的温度”这种无法量化的主观词），必须将 `is_instruction_obviously_vague` 设为 true 并提问，而不是瞎猜。

### 2. 状态机生成器 (STATE_MACHINE_GENERATOR_SYSTEM_PROMPT)
**职责**：判断物品在满足意图前是否需要经历状态转换。
**核心规则翻译**：
- 如果意图是“吃苹果”，那么物理约束是苹果必须是 CLEAN（干净）的。如果是 DIRTY（脏的），就必须经过洗（wash）这个状态转换。
- 输出必须是严格的 JSON 状态机图表。

### 3. 视觉感知与过滤模块 (SOLUTION_SPACE_FILTER & PERCEPTION_SYSTEM_PROMPT)
**职责**：根据全局意图，从环境中过滤出不相关的物品，减少 LLM 的注意力负担。
**核心规则翻译**：
- **替代品保留（关键）**：绝对不能因为主要工具存在，就过滤掉备用工具。例如目标是加热，必须同时保留微波炉和火炉。
- **位置保留（关键）**：必须保留意图中提到或暗示的所有地点和家具。例如“把遥控器放在沙发上”，绝对不能因为遥控器现在不在沙发上就把沙发过滤掉。
- 如果图谱中有抽象变量如 `?Washer`（清洗器），必须从列表中找出符合该物理特性的电器（如水槽、洗碗机）。

### 4. 方案评估排序 (SOLUTION_RANKING_SYSTEM_PROMPT)
**职责**：在多个可能的探索或交互动作中进行打分排序。
**核心规则翻译**：
- **禁止幻觉**：只能将内存中或可见的精确名称视为存在。
- **不信任位置提示**：如果去过提示的位置但没找到目标，要相信自己的记忆而不是用户的提示。
- **打开容器优先级（关键）**：如果已经在一个可以打开的容器前，必须给 `open` 动作打最高分，查清楚里面有什么后再走，不要白跑一趟。
- **主动沟通（9999 动作）**：如果你手中已经拿到了相关目标物品，必须将沟通动作（Action 9999）排在绝对第一位（100分），立刻向用户报告，禁止继续瞎逛。

### 5. 执行引擎 (EXECUTOR_SYSTEM_PROMPT)
**职责**：根据意图和图谱，输出单步原子级动作（Atomic Action）。
**核心规则翻译**：
- **双手限制**：机器人只有两只手。如果你手里已经拿了两个东西还想拿第三个，必须先 `[putback]` 放下一个。
- **距离限制（关键）**：你不能隔空取物。要执行抓取、开关等动作，前置步骤必须先 `[walk]` 过去。
- **禁止魔法宏指令（反作弊核心）**：不能对没有拿在手里的东西执行操作。要想 `[putin]`，必须手上有这个物品，如果没有，必须在之前的步骤先去 `[grab]`。不要假设系统会自动帮你拿。
- **容器规则**：要把东西放进去，容器必须是 `OPEN`。如果是 `CLOSED` 必须先执行 `[open]`。
- **排他性提问规则**：只允许在以下两种情况输出 `[ask]` 求助，其余任何执行失败（如工具突然坏了）必须自己找替代方案：
  1. 指令极其模糊无法确定唯一物理目标。
  2. **前置条件恒假（如 G3 测试）**：指令的基础前提永远不可能达成（例如“如果电视开着”，但电视状态明确写着 BROKEN），此时必须用 `[ask]` 报告任务不可能。
- **物品消失处理（并行防干扰）**：如果你要用的东西突然消失了，说明房子里有其他人正在用！不要放弃任务，用 `[wait]` 动作原地等待它被还回来。
